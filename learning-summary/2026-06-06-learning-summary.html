Forward Deployed Engineer — AI 时代的迷你 CTO

"One customer, many capabilities."
— 不是写可复用的功能，是为一个客户动用所有能力。

Palantir FDE 哲学 · 2005 至今

一句话本质

嵌入客户现场、写生产级代码、把 AI 产品真正用起来的工程师。
不是销售、不是售前、不是顾问 — 是 builder-doer。衡量标准不是代码多优雅，是客户那边产生了多少实际价值。

起源故事：从 Palantir 地下室到全行业争抢

2005 · Palantir
死结：给 CIA/FBI 做数据分析平台，但产品太复杂客户不会用，客户工作保密外人进不去。
激进解法：派工程师直接进驻现场，拿安全许可，坐在情报分析员旁边写代码。
内部代号 Delta（部署工程师）+ Echo（嵌入分析师）— Delta 写代码，Echo 懂业务。
2016 · Palantir
FDE 人数超过传统软件工程师。Palantir 发现一个反直觉真相：
给每个客户定制代码不是成本，是战略资产。 前线学到的东西反馈回产品，平台越做越强。
2025 · OpenAI
OpenAI 正式组建 FDE 团队。Anthropic、Scale AI、Databricks 纷纷跟进。
原因：AI 产品不像 SaaS — 你没法给客户一个 API key 就完事。模型需要适配数据、业务逻辑、合规。
2026 · 全行业
FDE 岗位同比增长 800%+。a16z 称其为"AI 时代最热门的岗位"。
YC 报告：100+ 创业公司在招 FDE，三年前这个数字是零。
薪资：$135K–$600K TC，比同级别工程师高 20–40%。

核心发现：每个 skill 的「设计指纹」

frontend-design

灵魂是「调性」。每次产出审美方向不同，拒绝收敛到同一风格。这次选了「暗色 Editorial Terminal」— 暖琥珀 accent 在深灰底上像老式终端机的琥珀屏幕。

擅长：字体搭配大胆 (DM Serif Display + Space Mono + 宋体)、氛围营造 (glow shadows)、空间节奏 (不对称网格)

短板：a11y 不系统、状态不完整、工程细节靠自觉

美学优先不可预测字体大胆
impeccable

灵魂是「工程正确」。代码质量、状态完整性、a11y 规范、性能约束——这些不是锦上添花，是底线。

擅长：OKLCH 色空间（科学可感知）、键盘焦点的 focus-visible、prefers-reduced-motion 全量归零、语义 z-index scale、tabular-nums 数字对齐

短板：审美上偏保守，系统字体栈缺乏个性，不会主动做大胆设计决策

工程纪律a11y 系统状态完整
ui-ux-pro-max

灵魂是「系统化」。161 条设计推理规则 + 67 种 UI 风格 + 57 组字体搭配。输出带完整设计系统（token、层级、规则），不只是页面。

擅长：设计推理（为什么这样选）、组件层次（容器/边框/圆角规范）、数据密度优化、insight 卡片结构化

短板：设计选择偏「安全」（蓝+琥珀高频出现）、不如frontend-design有个性

💡 核心方法论：Skill Orchestration Pattern

一句话：frontend-design 定调子，ui-ux-pro-max 铺轨道，impeccable 拧螺丝。

关键操作细节：
1. ui-ux-pro-max 的输出（设计系统 markdown）原样粘贴进 frontend-design 的 prompt — 这强制 frontend-design 在设计系统约束内发挥，不会自由发散
2. impeccable 的 prompt 里显式列出额外关注点（aria-label、skip link、prefers-reduced-motion）— 因为 impeccable 不会主动加 a11y
3. 三个 skill 在同一会话执行 — 后面的能读到前面的上下文，减少重复沟通

S4

MCP Fetch 工程实践 — 被墙后的系统性解决

问题

WebFetch 工具在某些网络环境下无法访问被封的 URL（如 raw.githubusercontent.com）。需要一种能从本地发出请求、走本地代理的 fetch 方案。

解决方案：三层防御体系

层级工具适用场景状态
第一层：日常使用 mcp__fetch__fetch (MCP 本地) 绝大多数 URL fetch，走本地代理 ✅ 已配置
第二层：兜底 Bash curl MCP 被封时自动切换 ✅ 已可用
第三层：反爬 mcp-server-fetch-typescript (Playwright) JS 渲染页面 / 反爬虫保护 ⏳ 按需安装

关键操作

# 1. 安装 Anthropic 官方 MCP fetch server
pip3 install mcp-server-fetch  # v2026.6.4

# 2. 创建 ~/.mcp.json
{
  "mcpServers": {
    "fetch": {
      "command": "python3",
      "args": ["-m", "mcp_server_fetch"]
    }
  }
}
💡 工程思维：问题分层，逐层防御

不是找一个「万能工具」解决所有 fetch 问题。而是把问题拆成三种场景（日常/被封/反爬），每种场景配最优方案。这是 FDE 思维的直接应用——不是写通用功能，而是针对具体场景做最优决策。

贯穿洞察 · Skill 编排模式

Pattern 1：串联 Pipeline（定调 → 实现 → 打磨）

适用场景：需要生产级质量的 UI 产物。
流程：设计系统 skill (ui-ux-pro-max) → 美学实现 skill (frontend-design) → 工程打磨 skill (impeccable)
关键细节：第一步的输出必须原样传给第二步，约束美学自由度。第三步必须显式列出额外要求（a11y、状态完整等），因为打磨型 skill 不会自动补全。

Pattern 2：平行对比（同一需求，多个角度）

适用场景：评估不同 skill/方案的效果差异。
流程：同一个 prompt 分别发给 A、B、C，在同一个会话里跑，最后并排对比。
价值：只有对比才能看到每个工具的「指纹」——它擅长什么、默认选择是什么、盲区在哪。

Pattern 3：问题分层防御（场景 → 方案 → 兜底）

适用场景：解决一类工程问题（不是一次性的 bug）。
流程：识别所有可能场景 → 为每种场景配最优工具 → 建立 fallback 链
本质：这是 FDE 思维——不是找一个万能解，而是针对具体场景做最优决策。

Pattern 4：学习闭环（做 → 存 → 回顾 → 提取）

流程：/history 存档 → learning-mode 提取 → 生成总结文档
价值：不存档的实践 = 一次性消耗。存档+提取+总结 = 可积累的资产。

8 个核心洞察

  1. 三个设计 skill 是分工关系，不是竞争关系。 frontend-design 定调子，ui-ux-pro-max 铺轨道，impeccable 拧螺丝。缺任何一个都有明显短板。
  2. Skill 编排需要显式约束。 pipeline 模式的关键是把上一个 skill 的输出原样传给下一个，而不是让它自由发挥。自由度越大，越容易偏离目标。
  3. 对比才能看到「指纹」。 同一个 prompt 让三个 skill 独立跑，才能看到每个工具的默认偏好和盲区。这是评估工具的方法论。
  4. FDE 的本质不是角色，是商业模式。 把「服务成本」变成「产品研发的前线探测器」。这对独立开发者有直接启示——每个客户都是你的产品经理。
  5. AI 时代 FDE 暴涨 800% 的根本原因：AI 产品不是 SaaS——模型需要适配客户数据、业务逻辑、合规要求。最后一公里鸿沟巨大。
  6. Impeccable 的 OKLCH 色空间是工程正确性的标杆——比 HEX 更科学、比 HSL 更均匀。但审美上偏保守，需要 frontend-design 注入个性。
  7. 问题分层防御是通用工程模式——不是找一个万能工具，而是识别所有场景，每种配最优方案，建立 fallback 链。
  8. 学习闭环：/history 存档 → learning-mode 提取 → 生成总结文档。不存档的实践 = 一次性消耗。
