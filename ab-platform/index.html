01 / 北极星

Landing 的核心命题

在最长的时间内保持各方对你的信心，直到平台跑出第一个正向飞轮。

大多数人一上来就列功能需求（分流、指标计算、显著性检验、报表），那是工程思维。业务思维的 landing 逻辑是反过来的：先找到谁最痛、痛在哪、为什么痛，再用平台去解决那个具体的痛。

进公司第一天的三个问题

不是技术选型，不是业务流程——是更根本的东西。

你要问的背后对应的为什么重要
哪些部门在做 AB 实验？你的用户图谱决定了谁是你的第一优先级
当前他们怎么做实验？现有工作流和替代方案Excel 手动？SQL 脚本？第三方工具？
卡点是什么？你的价值切入点谁最痛、痛在哪、为什么痛

访谈后发现的典型情况

角色怎么做实验最大痛点
算法侧自建小工具分流，内部包算分流逻辑有 bug，SRM 经常失败
运营侧SQL 写死对照组/实验组，手动 Excel 算每次 3 天出结果，太慢
产品侧不做实验，看环比拍板不知道怎么做，也不知道可信度

为什么前 30 天优先服务算法侧？

算法侧有「最苛刻的需求和最不能容忍的错误」——SRM 检验、分流一致性、高并发。把最难的场景跑通，其他场景平滑铺开。而且算法侧最不能容忍分流出错——如果分流有 bug，模型线上效果评估全偏，他们会逼你做到极致。

注意：算法实验频次高，但运营和产品的决策权重可能更大。一个定价策略实验决定几千万 GMV，一年可能只做几次。频次高不等于价值大。选择算法侧的核心逻辑是「最难的跑通了，其他的顺带就能覆盖」。

WEEK BY WEEK

02 / 分周作战

30 天详细路线图

节奏铁律：第 1-2 周只观察不承诺，第 3-4 周跑通最短闭环。第一个月不会有 UI 交付。为什么？因为你主动说出来，比你交不出来时让别人说你更有掌控感。

第 1 周 画地图 — 摸清生态
人 约谈三方
算法 TL：理解他们的实验全流程和现行痛点
数据工程 TL：摸清数据基建的边界
你的老板：对齐他对「成功」的定义
事 周五交付：现状诊断文档
长度不超过两页纸。包含：(1)各团队实验流程和痛点 (2)3 个月分阶段路线草图（不承诺细节，但让人看到节奏）(3)明确说明「第一个月不会有产出，我在建地基」
技术 对齐成功定义
老板的 OKR 里「实验基建」有人扛了——但「扛得好」的定义是什么？需要他自己说出来。你不需要猜。

关键对话 跟数据工程沟通时，别用业务语言。10 年分析师最容易踩的坑：一上来就用业务框架对话，对方不 care。你应该问：「现在你们最不想被上游改的是什么？」——问出他们的防御点，才能绕开。

第 2 周 嵌入算法团队 — 成为他们工作流的一部分
人 全程跟 1-2 个真实实验
从假设提出 → 分流设计 → 指标定义 → 数据收集 → 统计分析 → 业务结论。完整走一遍，不要打断、不要建议——只观察。
人 参加实验评审会
观察他们怎么决定一个实验上不上。谁拍板？看什么指标？什么情况下会推翻结论？——这是在理解他们的决策逻辑。
技术 梳理指标词典 V1
算法团队认可的 5-8 个核心指标。每个指标：口径定义 + 计算周期 + 去重逻辑。这是地基——口径错了，后面全错，而且别人一眼就能指出「你算错了」。
事 找到最烦的手工操作
现行方案里一定有一步让人咬牙切齿。大概率是某个手动 SQL 查询、某个 Excel 透视表、某个每次都要人工检查的分流均匀性——那就是你第一个要解决的问题。
第 3-4 周 最短闭环 — 一个真实实验跑通

前 30 天唯一要交付的东西：一个从分流到出结论的最短闭环，在算法团队一个真实实验上跑通。能跑过一个真实实验，比能跑一百个假想场景有用一万倍。

闭环三步
1. 分流确认——流量分两组，确认分流均匀（SRM 通过）
2. 指标计算——核心指标自动计算，出置信区间
3. 结论输出——一个结论页，能让人看完决定「该上线还是不上线」
技术 实现方式
不用系统——一个脚本 + 一份报告足够。数据链路可以用硬编码 SQL 先扛 30 天。统计正确性用标准方法先跑起来（t 检验、固定样本量、不窥探——够用）。
事 成功标准
算法 TL 看完报告说「比我之前快/准」。他能指着结果说「这是我们新平台产出的」——所有权明确，功劳才不模糊。
TRAPS

03 / 三大陷阱

Landing 失败的三种经典死法

之前见过别人 land 失败也好，自己踩过坑也好——这三个坑之间有因果关系。

事铺太大 → 迟迟不出活 → 各方失去耐心 → 互相推诿 → 全盘否定历史 → 恶性循环
层面典型死法为什么发生如何避免
人 无达成共赢目标，汇报上互相推诿 没有提前对齐各方对「成功」的定义 第 1 周跟老板和相关 TL 明确：什么叫第一阶段的成功？三个月后我们看什么指标判断这个平台成功了？
事 铺得过大，周期过长 企图一步到位做全功能平台，而不是先验证最小价值 砍到只交付一个实验的最短闭环。分流 + 指标 + 结论 = 够了。其他功能等有人真的需要再加。
技术 全盘否定历史沉淀，全部焕新 为了追进度，觉得从头来最快——但实际上引入新 bug、丢掉已有经验 先理解历史方案为什么长成那样。算法团队自建的小工具虽然简陋，但里面藏着他们最在意的需求。

额外警惕：10 年分析师最容易犯的错 —— 太懂业务，一上来就用业务语言跟数据工程对话。对方不 care 你的分析框架，他们 care 的是：这个人的需求会不会频繁变？会不会提不合理的计算量？会不会出了问题先甩锅给数据源？前两周跟数据工程沟通，最重要的一个问题是：「你们最不想被上游改的是什么？」

DECISIONS

05 / 基础设施

第 2-3 个月：三个不可逆决策

先别碰 UI。最容易犯的错误是第 5 周就开始画原型图。先定这三个底层决策再写代码。点击展开每个决策的完整推演。

01
最不可逆 — 定下去就改不了
分流层
用户标识选 user_id 还是 device_id？这个决策影响所有后续实验的分流一致性和用户追踪。选错之后换不了。
user_id vs device_id：user_id 能跨设备追踪用户，但未登录用户无法分流；device_id 覆盖面全，但用户换设备就丢失。需要根据业务场景选择——如果大部分交易需要登录，user_id 更可靠。
分流算法：hash(实验ID + user_id) % 100。简单、可验证、无中心化依赖。任何一个工程师都能独立验证分流是否正确。
互斥与层域：多个实验同时跑，要不要设计层？一期可以先不做——除非算法团队已经在同时跑 5+ 个实验且互相干扰。做层的成本是复杂性，不做层的成本是流量冲突，权衡点在于实验并发数。
常见 bug 来源：hash 函数版本差异、上游缓存未刷新、灰度发布时新旧分流逻辑不一致、实验组页面加载慢导致实际进组人数缩水——这就是为什么 SRM 检验不能省。
02
地基 — 口径错了后面全错
指标层
前 30 天必须死磕指标口径。算法团队说「转化率」、运营说「下单率」、产品说「支付转化率」——可能是同一个东西，也可能不是。三种计算方法会得到三种不同结果，如果在实验平台上看到数字跟预期不一样，用户第一反应是「平台算错了」。
核心指标 5-8 个，从算法侧开始：CTR、CVR、人均 GMV、客单价、留存率、加购率、复购率。每个指标必须文档化：口径定义、计算周期、去重逻辑。
日级更新即可：一期不要做实时。实时指标需要流式计算架构，复杂度翻倍，价值有限。日级 T+1 够用。
数据链路依赖：指标计算不是你的代码算出来就完了。上游有埋点、数仓 ETL、BI 层。你依赖别人，别人的排期不受你控制。前 30 天用 SQL 硬扛，同时把正式链路需求提给数据工程——给他们充足的排期时间。
绝对值和比例指标分开处理：销售额（绝对值）的方差用 s²/n，转化率（比例）的方差用 p(1-p)/n。虽然数学上两者趋同，但低转化率场景下标准误远超业务想检测的效应量——你能检测到的最小效应受到样本量制约。
03
可迭代 — 先用标准方法跑起来
统计层
一期不做序贯检验、不做 CUPED、不做因果森林。用固定样本量 + t 检验 + 卡方检验 + SRM + 置信区间。这些方法的缺陷在业务决策中造成的损失，远小于你前三个月没交付任何一个可用结果造成的信任损失。
一期统计工具箱：固定样本量设计 + t 检验（连续指标）+ 卡方检验（比例指标）+ SRM 检验 + 置信区间展示。这些够用 80% 的场景。
二期补充：CUPED（用实验前数据降方差，可缩减 10-50% 方差）、序贯检验/Alpha-spending（允许安全窥探）、因果森林异质性分析（找对策略反应最大的用户群）。
结论页设计原则：永远不要把点估计单独展示给决策者。必须同时展示：(1)点估计+置信区间 (2)SRM 是否通过 (3)样本量是否达标。三个问题：指标变了吗？结果可信吗？该上线吗？
窥探效应的坑：每天看一次 p 值、显著了就停——这会让假阳性率从 5% 飙升到约 40%（看 10 次的话）。一期通过流程约束禁止窥探——定好样本量和实验时长，跑完再看。二期再上技术方案（序贯检验）。
STATISTICS

06 / 统计基础

结论页背后的方法论

你的平台不管界面多好看，最终只回答三个问题。以下是每个问题背后的统计工具。

问题一：指标变了吗？—— 点估计 + 置信区间

点估计就是你的一个数字结论：「实验组人均 GMV 比对照组高 3.2 元」。3.2 就是点估计。它能回答方向上谁赢了、赢了多少，但不能回答这个 3.2 稳不稳。

置信区间就是稳不稳的度量。标准误 = 0.5 元，则 95% 置信区间 = 3.2 ± 1.96 × 0.5 ≈ [2.22, 4.18]。置信区间的意思是：如果重做一次实验，差异 95% 概率落在这个区间里。

置信区间长这样意味着你该怎么做
[2.22, 4.18]下限 > 0，全体大于零大概率真有效，上线
[-0.8, 7.2]下限 < 0，包含零可能有效也可能无效，加大样本
[-3.1, -0.5]上限 < 0，全体小于零大概率有负效果，放弃

铁律：永远不要把点估计单独展示给决策者。必须带置信区间。 只看点估计就做决策 = 闭眼下注。

问题二：结果可信吗？—— SRM 检验

SRM（Sample Ratio Mismatch）= 样本比例不匹配。你设计 50:50 分流，实际跑了 10000 个用户，实验组 5100，对照组 4900。这合理吗？还是分流系统出 bug 了？SRM 检验用卡方检验判断：观测到的两组人数比例，和设计的比例，差异是否在随机范围内。

分流 Bug 类型具体表现不检验的后果
hash 函数版本差异不同服务节点算出来的分组不一致同一用户在不同请求中被分到不同组——数据污染
上游缓存未刷新用户被分到某一组后，缓存锁定，实验条件变更无效实验组实际样本缩小，统计功效不足
灰度发布分流不一致新旧版本代码的分流逻辑有差异不可察觉的系统偏差
实验组页面更慢实验组加载慢导致更多用户流失，实际进组人数缩水你以为转化率更高是因为策略好，实际是因为留下了高质量的、网速快的用户——幸存者偏差

SRM 通过 → 你的分流是干净的，两组可比。SRM 不通过 → 先别分析指标，先修分流。

方差估计——四个实际会出错的场景

方差决定你「能不能信这个实验结果」。方差估小了 → 该放弃的策略上了线，浪费资源。方差估大了 → 该上线的策略放弃了，错失增长。

类型根因表现解决方案
A. 指标类型 把 0/1 比例指标当连续值处理，或用错分布假设 置信区间系统性地偏窄或偏宽。对于 0/1 数据，s²/n 和 p(1-p)/n 实际上趋同——真正的问题不是公式，而是低转化率时标准误本身就很大 低转化率（< 1%）时，你能检测到的最小效应量受样本量制约。样本翻 4 倍，标准误才减半
B. 用户不独立 社交电商、分享裂变、同一家庭/公司的用户互相影响 你以为 10000 个用户 = 10000 个独立信号源，但实际有效信号源远小于此。你低估了方差，置信区间偏窄，假阳性率升高 聚类标准误（cluster-robust SE），以用户圈/城市/时间段为聚类单位重新估方差
C. 窥探效应 每天看一次 p 值，哪次显著了就停实验宣布胜利 看 10 次，即使策略完全无效，至少一次 p < 0.05 的概率 ≈ 40%。你发现的有效策略上线后效果蒸发——因为你发现的不是因果效应，是多看几眼碰到的运气 一期通过流程约束：定好样本量和实验时长，跑完再看。二期上序贯检验（Alpha-spending），每次窥探消耗一点 α 预算
D. 离群值主导 极端用户（土豪大单、刷量账号）拉偏均值 前 9 天差 1 个点，第 10 天来个大单，差变成 8 个点。方差爆炸，结果极不稳定。同时 s²/n 也被带偏——给你一种「方差很大、不显著」的错觉 CUPED（用实验前数据降方差）、Winsorization（截尾）、或用中位数替代均值估计

实验指标方差计算一览

指标类型例子数据特征方差公式注意事项
绝对值指标 销售额、GMV、客单价 连续值，每个用户一个数值 Var = s² / n
s² = Σ(Yᵢ−Ȳ)²/(n−1)		 s² 从数据里算出来，用户之间差异越大 → 方差越大 → 指标越不稳
比例指标 转化率、点击率、留存率 0/1 变量，每个用户要么是 1 要么是 0 Var = p(1−p) / n p 接近 0.5 时方差最大，p 接近 0 或 1 时方差很小。方差不需要从数据估——p 本身就告诉你方差

两个公式本质是同源的：均值的方差 = σ²/n。对绝对值指标，σ² 用 s² 从数据估；对 0/1 数据，σ² 天然等于 p(1-p)。所以你把转化率用 s²/n 去算，数学上也对——s² 会自动约等于 p(1-p)。

07 / 进公司前

三件事现在就能做

  • 写一份你自己的指标词典草稿。电商核心指标——转化率、GMV、客单价、复购率、留存率、加购率——你对每个指标的定义、分层维度、计算周期是什么？带进去直接对齐，不用等别人告诉你。跟算法团队对齐时能用业务语言沟通，跟数据工程对齐时能转化为字段级需求。
  • 准备 3 个「我从零搭建实验体系踩过的坑」的故事。不是讲你有多了不起，是讲你踩过什么坑、怎么发现的、怎么修的。面试可能没讲细节，入职后跟算法团队聊，这些才建立信任——因为你证明了你理解他们的世界、知道什么事会出问题。
  • 选一个开源工具先跑一遍 demo。GrowthBook 或 Wasabi，本地 Docker 拉起来，跑一个假实验。不需要用它，但你会知道一个成熟实验平台有哪些模块——分流、指标定义、统计检验、报表——不至于从零画图纸。同时拿到一张关键人物清单：第一周要见的人、每个人的 OKR 大概是什么、他最怕什么、你能给他什么。
PRINCIPLES

08 / 三条原则

Landing 成功的底层逻辑

P

People — 找到 30 天价值最大化的合作方，建立信任，通过交付维持信任

选算法侧不是因为频次高，是因为他们有最苛刻的需求和最不能容忍的错误。把最难的人服务好，其他人的需求是降维覆盖。

信任不是一次汇报建立的——是算法 TL 主动向别人推荐你的平台才算建成。在此之前，你只是在证明自己。在此之后，你才有了真正的内部盟友。

W

Work — 跑通一个最短闭环，而不被铺大的冲动带走

能跑过一个真实实验，比能跑一百个假想场景有用一万倍。前 30 天唯一要交付的，是算法团队一个真实实验从分流到结论的完整链路。

不是流程本身重要——是你能指着结果说「这是我建的平台产出的」。所有权明确，功劳才不模糊。闭环的终点不是「程序跑通了」，而是「算法 TL 用它做了一个实验决策」。

T

Tech — 找到不可逆决策，先定再动，统计方法可以迭代

分流层的用户标识是最不可逆的——user_id 还是 device_id 定下就改不了。指标口径是地基——口径错了后面全错。统计方法一期用 t 检验就够，后续再补 CUPED、序贯检验、因果森林。

不要在第 5 周开始画 UI 原型。前三个月把分流、指标、统计三层做对，结论页可以简陋到一张表格——但里面的数字必须是对的、可信的、能指导决策的。

AB 实验平台的本质不是软件系统，是决策基础设施。你交付的不是功能列表，是可复用的实验方法论。
